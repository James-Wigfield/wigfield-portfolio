Pseudo-Random Number Generation Lab 1

Explore how the Linear Congruential Generator (LCG) produces deterministic sequences that mimic randomness, and why parameter choice is critical.

∑ Mathematical Foundation
Core LCG Recurrence Relation
$$X_{n+1} = (a \cdot X_n + c) \bmod m$$
$X_0$Seed — initial value; determines reproducibility of the sequence
$a$ Multiplier — scales the current state; must be chosen carefully
$c$ Increment — added after multiplication; use $c=0$ for multiplicative generators
$m$ Modulus — upper bound on output; output is always in $[0,\ m-1]$
Mersenne Primes (ideal moduli)
$$m = 2^n - 1, \quad n \in \{3, 5, 7, 13, 17, \ldots\}$$
Why Mersenne primes? A modulus with many factors (e.g., 12 = 2²×3) causes the LCG to cycle through only a subset of values — poor equidistribution. Mersenne primes like 7 (2³−1) and 31 (2⁵−1) have very few factors, maximising the period and producing more uniform outputs. The lab uses modulus 31 as its default.
Two-Stage PCG (NumPy's default generator)
$$\text{PCG-64}: \quad \underbrace{X_{n+1} = (a \cdot X_n + c) \bmod 2^{64}}_{\text{LCG core}} \xrightarrow{\ \text{permute}\ } \text{output}$$

The Permuted Congruential Generator (PCG) composes an LCG with an output permutation function, dramatically improving statistical quality while retaining the simplicity and speed of an LCG.

Full-Period Condition (Hull–Dobell Theorem)
$$\text{Period} = m \iff \gcd(c, m)=1,\ (a-1) \text{ div by all prime factors of } m,\ 4 \mid (a-1) \text{ if } 4 \mid m$$

Bin Packing Heuristics Lab 2

Compare four packing strategies on the Crest Packing Problem. Items arrive sequentially on a conveyor belt and must be placed immediately.

📦 Strategies & Complexity
Crest Packing Problem (Lab 2): Items of scale factors in [0.1, 1.0] arrive on a conveyor. Pack into unit-capacity boxes. Items cannot be skipped or reordered. The lab baseline greedy score is 663 / 2000 (20 conveyors × 100 items).
First Fit (FF) O(n²)
Scan bins from the start; place in the first bin that fits. Opens a new bin only if no existing bin has space.
Next Fit (NF) O(n)
Only check the current bin. If item doesn't fit, close it and open a new one. Never revisits old bins — fastest but wastes space.
Best Fit (BF) O(n²)
Place in the bin with least remaining capacity that still fits — the tightest fit. Reduces wasted space per bin.
First Fit Decreasing (FFD) O(n log n)
Sort items in decreasing order first (requires all sizes upfront), then apply First Fit. Offline only — cannot be used on the conveyor directly.
Capacity Constraint
$$\sum_{i \in \text{box}_k} s_i \leq C = 1.0 \quad \forall\ k$$
Sorted Insertion via Binary Search
$$\text{bisect.insort}(L, x) \implies O(\log n)\ \text{search} + O(n)\ \text{insert}$$

Python's bisect module enables $O(\log n)$ search in sorted lists, useful for maintaining sorted structures during packing.

Job Shop Scheduling Problem Labs 3–5

Assign operations to machines to minimise makespan. Edit the instance, compute a greedy schedule, and inspect the Gantt chart.

📋 Formal Formulation
JSSP: Given $n$ jobs and $m$ machines, each job requires $m$ operations in a fixed sequence. Each operation $O_{j,k}$ must run on machine $\mu_{j,k}$ for $p_{j,k}$ time units. Schedule all operations to minimise total completion time.
Objective — Minimise Makespan
$$C_{\max} = \max_{j=1}^{n}\, C_j = \max_{j,k}\, \bigl(S_{j,k} + p_{j,k}\bigr)$$

Feasibility Constraints

  1. Machine capacity: $S_{j,k} \geq S_{j',k} + p_{j',k}$ or vice-versa for any $j \neq j'$ on same machine
  2. Precedence: $S_{j,k+1} \geq S_{j,k} + p_{j,k}$ — each job's operations are ordered
  3. Assignment: Operation $O_{j,k}$ must run on machine $\mu_{j,k}$
  4. Non-negativity: $S_{j,k} \geq 0\ \forall\ j, k$
  5. Completion: $S_{j,k} + p_{j,k} \leq C_{\max}\ \forall\ j, k$
Internal Representations (from Lab 3)
$$\text{Instance} \in \mathbb{Z}^{n \times m \times 2} \quad \Bigl([\mu_{j,k},\ p_{j,k}]\Bigr)$$
$$\text{Schedule} \in \mathbb{Z}^{m \times n \times 3} \quad \Bigl([\text{job},\ S_{j,k},\ S_{j,k}+p_{j,k}]\Bigr)$$
Two-Level Feasibility Check (Lab 3)
consistent(): No two operations overlap on the same machine; no job's operations overlap — checks internal coherence only.

satisfies(instance): Calls consistent(), then additionally checks machine assignments match the instance and precedence constraints are respected.

Solution Space Analysis Labs 4–5

Visualise the combinatorial explosion that makes exhaustive enumeration infeasible and motivates random and heuristic search.

📊 Combinatorial Explosion
Solution Space Size
$$|\mathcal{S}| = (n!)^m$$
Lab 4 key insight: Each machine independently orders $n$ jobs — giving $n!$ orderings per machine. Across $m$ machines, the total candidate count is $(n!)^m$. For the classic ft10 benchmark (10 jobs, 10 machines), this is $(10!)^{10} \approx 3.6 \times 10^{65}$.

Lab 5 — Unit-Time Simplification
Lab 5 uses unit-time JSSP ($p_{j,k} = 1\ \forall j,k$) to make enumeration tractable for small instances. This gives a lower bound on makespan: $C_{\max} \geq n$ (each machine must process all $n$ jobs, each taking 1 unit). For a 3×3 unit-time instance, the minimum possible makespan is 3.

Beyond Exhaustive & Random Search:

→ Greedy construction — shortest processing time (SPT), longest (LPT)
→ Local search — swap adjacent operations on critical path
→ Evolutionary algorithms — crossover & mutation over permutations
→ Branch & bound — prune provably suboptimal subtrees
→ Tabu search — neighbourhood search with memory to avoid cycles
Complexity Classification
$$\text{JSSP} \in \mathcal{NP}\text{-hard} \quad (\text{even for } n = 3,\ m = 3)$$
